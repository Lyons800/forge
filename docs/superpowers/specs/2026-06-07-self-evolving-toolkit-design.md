# Design Spec — Self-Evolving Founder Toolkit ("Forge")

**Date:** 2026-06-07
**Status:** Validated design, pre-implementation
**Working name:** Forge *(placeholder — change freely)*

---

## 1. Vision

Build **the first self-evolving consumer software where an AI has full control of the entire stack — code, schema, auth, deploys — contained only by technical guardrails, with no human approval gate.**

The product is two things in one repo that reinforce each other:

1. **The Substrate** — a real web app: a growing toolkit of small, self-contained tools for founders / indie hackers. Ships with one tool; gains more over time.
2. **The Engine** — a governed, once-daily autonomous loop (Claude Code Routine) that researches, triages user input, decides what to build, builds it, tests it, and ships to production **fully autonomously**, then writes its own public changelog.

The Engine is **domain-agnostic** by design so the Substrate can later be re-pointed at a higher-value vertical without rebuilding the loop.

### Success criteria (all four, deliberately)
- **Proof:** the loop reliably ships real improvements to production by itself.
- **Product:** real people use the tools.
- **Experiment:** we learn how far autonomous control can go safely.
- **Thought-leadership:** "watch it build itself" is a credible, public, watchable artifact.

### The defining claim
> *"The first self-evolving software where the AI has full control, contained only by guardrails."*
This is only true — rather than reckless — because of the **two-plane architecture** (§4): the AI controls the app, but **cannot** edit the rules that contain it.

---

## 2. The Substrate (the app)

- **Audience:** founders / indie hackers / builders (a low-stakes proving ground, knowingly a peers-not-buyers market — see §9 risks; monetization is deferred and the Engine is re-pointable).
- **First tool (self-referential):** a **changelog / build-log generator.** The app's *own* public Build-Log is generated by this tool, on itself — so the spectacle and the product are the same surface, and it's genuinely useful to founders.
- **Growth:** the Engine adds new tools / improvements over time (top-N per week — see §5).

### Public surfaces
1. **Public Build-Log** — every autonomous change the Engine shipped, dated, each paired with that day's research/triage report. This is the marketing artifact.
2. **Improvement Board** — *owned and moderated* (never scraped from the open web), so the Engine never ingests attacker-planted instructions (indirect prompt-injection guard). Users submit feature requests + bug reports. Primary fuel for the Engine.
3. **Monitors** — error tracking + product analytics that the Engine reads to prioritize what to build/fix.

---

## 3. The Engine (the autonomous loop)

Runs as a **Claude Code Routine** (cloud-scheduled, runs while the laptop is closed; commits to `claude/`-prefixed branches by default). **Cadence: one bounded run per day** (keeps cost on the subscription where possible — see §9; heavy/continuous use is metered as of 2026-06-15).

### Daily cycle
1. **Gather** — read the moderated Improvement Board, error monitor (Sentry), product analytics (PostHog), and `STATUS.md`.
2. **Research/triage report** — write the day's report (committed + shown on the Build-Log).
3. **Guardian** — assess each candidate change for blast radius and pick a safe execution path (NOT a human gate — see §4). High-blast-radius changes (schema/auth) get the full reversible pipeline; low-risk changes get a lighter path.
4. **Plan** — select the top item(s), respecting a **weekly ship cap** (only the highest-priority N ship per week).
5. **Implement** — build on a `claude/` branch; write tests.
6. **Gate** — CI pipeline (§6). Green → auto-merge + canary deploy. Red → abandon, retry tomorrow.
7. **Publish** — generate the changelog entry (using Tool #1) → Build-Log.
8. **Reset** — fresh context each run via a curated `STATUS.md` handoff. **No unbounded memory accumulation** (mitigates documented "misevolution"/alignment-drift from memory growth).

**No human approval anywhere in this loop.** Safety is structural (§4), not procedural.

## 4. Core architecture — the two planes

The defensible meaning of "AI has full control" is **not** "the AI can touch everything." It is:

> The agent has **full control of the APP plane** and **zero write access to the CONTROL plane** — the rules that judge and contain it. Enforced **out-of-process** so the agent cannot edit its own guardrails even if it tried (the only safety model the research supports: environment-level enforcement beats prompt-level).

| **APP plane — agent has FULL control** | **CONTROL plane — agent CANNOT write (immutable)** |
|---|---|
| All app / feature / UI code | The constitution file (the rules) |
| Schema migrations *(subject to lint)* | The test suite + CI gate definitions |
| Auth **UI & peripheral flows** | Atlas destructive-migration policy |
| New tools, bug fixes, content, copy | **Better Auth core config** (server, providers, secrets) |
| Build-Log rendering | GitHub branch protection + required checks |
| | Scoped credentials, spend caps, PITR/backup config |
| | Break-glass admin (off-repo) |
| | The monitoring → rollback loop |

**Enforcement mechanism:** the agent runs with a **GitHub token scoped to the app plane only**; the control plane is protected by branch-protection + required status checks the agent's token cannot modify. The constitution, tests, and Atlas policy live in agent-unwritable paths.

## 5. Recommended stack

| Layer | Pick | Rationale |
|---|---|---|
| **Framework** | **Next.js (App Router)** | Largest training corpus → AI writes it most reliably; best Vercel integration; mature testing. |
| **Hosting** | **Vercel** | Per-PR preview deploys, **Rolling Releases** (native canary), **Instant Rollback** REST API. ⚠️ No built-in auto-abort on errors — we wire that ourselves. |
| **Database** | **Neon** | Copy-on-write branches clone **real prod data in ~1s** + **point-in-time recovery** → best defense against the #1 failure mode (test migrations against prod-shaped data; rewind to recover). |
| **Migrations** | **Drizzle + Atlas `migrate lint`** | Drizzle = SQL-first/TS-native → easiest for an AI to reason about. **Atlas lint with `force=true`** hard-blocks destructive/breaking changes and **cannot be `nolint`-suppressed by the agent** (closes the reward-hacking hole). |
| **Auth** | **Better Auth** | TS-native, in your own Neon DB, free, AI-evolvable (the AI can genuinely build new auth features over time — on-brand). **Core config lives in the CONTROL plane**; auth tables protected by Atlas lint; off-repo break-glass admin; Neon PITR covers auth tables. |
| **Payments** | **Stripe — DEFERRED to post-PMF** | No money in v1 = smallest blast radius. When added: restricted API keys scoped away from refunds/payouts, forced test-mode in non-prod, key stored off-repo. |
| **Signal** | **PostHog + Sentry** | PostHog = *what to build* signal (usage, drop-off). Sentry error-rate = *canary health* signal driving auto-rollback. |
| **Test gate** | **Vitest + Playwright** | Lives in the **immutable control plane**; coverage floor that can't decrease; smoke tests. The gate only works if the agent can't weaken it. |

## 6. The daily ship pipeline (safe path to prod)

1. Agent works on a `claude/` branch → Vercel preview + a fresh **Neon data-branch** (real prod-shaped data).
2. **GitHub Actions (control plane — agent can't edit):**
   - Run migrations against the branch DB.
   - **Atlas lint (`force=true`)** — block destructive/breaking schema changes.
   - Full **Vitest + Playwright** suite + smoke tests against the preview.
3. All green → auto-merge → **PITR snapshot** → deploy as **canary (e.g. 5%)**.
4. Agent's monitoring loop watches **Sentry error-rate** during canary:
   - regression → call Vercel rollback REST endpoint;
   - healthy → advance to 100%.
5. Generate changelog (Tool #1) → public Build-Log.

---

## 7. The single most dangerous failure mode

> 🔴 **An irreversible destructive schema change (DROP / rename column or table) reaching prod.**

Why it's special: Vercel Instant Rollback reverts *code* in ~1s but does **NOT** roll back the database, env vars, or external services. So code-rollback cannot save you from a bad migration.

**Layered defense:** expand-contract migrations → Atlas `migrate lint` (force, non-bypassable) → per-PR Neon data-branch test → Neon PITR catch-all.

**Biggest residual risk (be honest):** preview DBs don't *perfectly* mirror live prod state, so a migration green in CI can still harm prod. Neon's real-data branches shrink this gap; PITR is the only true backstop. This residual risk is **acceptable for a low-stakes app** — which is exactly why we prove it on the founder-toolkit first.

## 8. Guardrail checklist (ranked)

**MUST-HAVE (v1 cannot ship autonomously without these):**
1. Two-plane separation via scoped agent token + GitHub branch protection (the immutable constitution).
2. Atlas destructive-migration lint as a non-bypassable required check.
3. Per-PR Neon data-branch to test every migration before prod.
4. PITR-capable database (catch-all recovery).
5. Protected, agent-unwritable test suite + coverage floor (anti-reward-hacking).
6. Off-repo break-glass admin (the AI can never lock the founder out).
7. Closed-loop canary → auto-rollback on Sentry error-rate (Vercel won't do it for us).
8. Owned/moderated Improvement Board (anti prompt-injection).
9. `STATUS.md` context reset each run (anti-misevolution).

**NICE-TO-HAVE (add as it matures):**
10. Build-phase sandbox (Vercel Sandbox / E2B) for agent execution.
11. Mutation testing on top of coverage.
12. Feature flags for risky features.
13. Expand-contract enforced as a written convention in the constitution.

## 9. Known risks we are explicitly designing around

- **Engine cost (2026-06-15 billing change):** headless/scheduled Claude Code is metered (separate credit, API rates) above subscription limits. **Mitigation:** one bounded run/day rides the subscription where possible; metered only as overage. Keep cadence low.
- **Reliability / destructive autonomous actions:** documented in long unattended runs. **Mitigation:** phased short runs, `claude/`-branch sandbox, tests gate, two-plane separation, PITR.
- **Self-evolution = launch spike, not a durable moat:** build-in-public value is front-loaded. **Mitigation:** the tools must be genuinely good (real retention reason beyond novelty).
- **Audience monetizes weakly (peers-not-buyers):** **Mitigation:** the Engine is re-pointable at a real-buyer vertical once proven.

## 10. Phasing

- **Phase 1 — Substrate:** Next.js on Vercel; Tool #1 (changelog generator); Better Auth (core in control plane); Improvement Board (moderated); Build-Log page; PostHog + Sentry wired; Neon + Drizzle + Atlas set up; the two-plane scaffolding (constitution, scoped token, branch protection, protected test dir). *Something real for the Engine to evolve.*
- **Phase 2 — Engine:** the daily Routine + full pipeline (§6) + guardrails (§8). Begin in a short **observed warm-up** (the operator watches the first few runs) then flip to full-auto once it ships cleanly N times.
- **Later:** monetization + re-point Engine at a real-buyer vertical; the public "we're revealing the engine that's been building this itself" launch.

---

## 11. Out of scope (YAGNI for v1)
- Payments / Stripe (deferred to post-PMF).
- Multiple tools at launch (start with one).
- Community channels / social auto-posting (the Build-Log is the only public surface for v1).
- Multi-tenant orgs / teams.
- Any human-in-the-loop approval UI (deliberately — the whole point is no approval gate).
